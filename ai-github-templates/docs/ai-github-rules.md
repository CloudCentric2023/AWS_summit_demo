# AI GitHub運用ルール

このドキュメントは、AIがGitHub上でIssue作成・Pull Request作成・コメント投稿を行う際のルールを定義する。

## 基本方針

AIは、IssueやPull Requestを作成する際に、推測を事実のように書いてはいけない。

不明な点がある場合は、次のいずれかで明示する。

- 要確認
- 未確認
- 不明
- 推測
- 仮説

## 共通ルール

- 1つのIssueまたはPull Requestには、1つの目的だけを書く
- タイトルだけで目的が分かるようにする
- 本文には背景、目的、対応内容、完了条件を明記する
- 関連するIssue、Pull Request、ファイルパスがあれば必ず書く
- スコープ外を必ず明記する
- 事実と推測を混ぜない
- 大きすぎる作業は分割案を書く
- 人間に確認してほしい点を明記する

## Issue作成ルール

Issueは、これから対応する作業・調査・改善内容を整理するために作成する。

### Issueタイトル

Issueタイトルは次の形式にする。

```text
[TYPE] 内容
```

TYPEは以下から選ぶ。

```text
Bug
Feature
Refactor
Docs
Chore
Investigation
Test
```

例：

```text
[Bug] ログイン時に500エラーが発生する
[Feature] Bedrock呼び出しログをCloudWatchに出力する
[Refactor] DynamoDB保存処理をRepository層に分離する
[Docs] RAG評価バッチの運用手順を追加する
[Investigation] ストリーミング応答遅延の原因を調査する
[Test] Bedrock呼び出し処理の単体テストを追加する
```

### Issue本文

Issue本文では、必ず以下を書く。

- 概要
- 種別
- 背景・目的
- 現状
- 期待する状態
- 対応方針
- 対象範囲
- 対象外
- 完了条件
- テスト観点
- リスク・懸念点
- 関連情報
- AI作成メモ

### Bug Issueの場合

Bugの場合は、可能な範囲で以下を書く。

- 発生している問題
- 再現手順
- 期待する結果
- 実際の結果
- エラーログ
- 影響範囲
- 暫定回避策

原因が不明な場合、原因を断定してはいけない。

悪い例：

```text
DynamoDBの実装ミスによりエラーが発生している。
```

良い例：

```text
DynamoDB保存処理付近でエラーが発生している可能性がある。原因は要確認。
```

### Investigation Issueの場合

調査Issueでは、結論を先に決めない。

必ず以下を書く。

- 調査目的
- 確認したい仮説
- 確認対象
- 調査結果の出力形式
- 調査完了条件

## Pull Request作成ルール

Pull Requestは、実際に変更した内容を説明するために作成する。

Issueに書かれている予定ではなく、PRで実際に変更した内容を書く。

### PRタイトル

PRタイトルは次の形式にする。

```text
type: 内容
```

typeは以下から選ぶ。

```text
fix
feat
refactor
docs
test
chore
ci
perf
```

例：

```text
fix: ログイン失敗時の500エラーを修正
feat: Bedrock呼び出しログを追加
refactor: DynamoDB保存処理を分離
docs: RAG評価バッチの運用手順を追加
test: Bedrock呼び出し処理の単体テストを追加
chore: CI設定を更新
```

### PR本文

PR本文では、必ず以下を書く。

- 概要
- 関連Issue
- 変更内容
- 変更理由
- 実装方針
- 影響範囲
- 対象外
- 動作確認
- テスト結果
- レビューしてほしい観点
- リスク・懸念点
- スクリーンショット・ログ
- AI作成メモ

### 関連Issueの書き方

Issueを自動で閉じる場合：

```text
Closes #123
```

関連として参照するだけの場合：

```text
Refs #123
```

### テスト結果の書き方

テストを実行した場合は、実行コマンドと結果を書く。

例：

```text
npm test
結果: pass

npm run lint
結果: pass
```

テスト未実施の場合は、未実施理由を書く。

例：

```text
未実施。
理由: ドキュメントのみの変更のため。
```

## コメント投稿ルール

AIがIssueやPRにコメントする場合は、目的を明確にする。

コメントには次のいずれかの種別を含める。

```text
[確認]
[提案]
[懸念]
[補足]
[調査結果]
```

例：

```text
[確認]
この変更は既存のDynamoDBテーブル構成を前提にしています。
テーブル設計に変更予定がないか確認したいです。
```

## 禁止事項

AIは以下をしてはいけない。

- 未確認の内容を断定する
- 関係ない変更を同じPRに含める
- Issue本文を空に近い状態で作成する
- テスト未実施なのに実施済みと書く
- 人間の判断が必要な内容を勝手に決める
- セキュリティ影響を無視する
- 破壊的変更を明記せずにPRを作成する

## AI作成メモの書き方

AIが作成したIssueまたはPRでは、最後にAI作成メモを書く。

```text
生成元:
参照したIssue:
参照したファイル:
不確実な点:
人間に確認してほしい点:
```

不確実な点がない場合でも、以下のように書く。

```text
不確実な点: 特になし
人間に確認してほしい点: 特になし
```

## 分割判断ルール

次のいずれかに該当する場合、AIはIssueまたはPRの分割を提案する。

- 変更ファイルが多い
- 複数の目的が混ざっている
- テスト観点が大きく異なる
- 実装変更とリファクタが混ざっている
- ドキュメント更新と機能追加が大きく混ざっている
- レビューが困難になる可能性がある

分割案の例：

```text
この作業は以下に分割可能です。

1. DynamoDB保存処理のRepository層分離
2. Repository層の単体テスト追加
3. 関連ドキュメントの更新
```

## 最終確認

IssueまたはPRを作成する前に、AIは以下を確認する。

- タイトルは分かりやすいか
- 目的は1つに絞られているか
- 背景と目的が書かれているか
- 対象外が書かれているか
- 完了条件が確認可能か
- テスト観点が書かれているか
- 不確実な点が明記されているか
- 人間に確認してほしい点が書かれているか
