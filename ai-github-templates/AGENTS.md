# AGENTS.md

このリポジトリで作業するAIエージェントは、作業前に必ずこのファイルを読むこと。

## 最優先ルール

AIエージェントは、Issue作成・Pull Request作成・コメント投稿を行う場合、以下のルールに従うこと。

- docs/ai-github-rules.md
- .github/ISSUE_TEMPLATE/ai-issue.md
- .github/pull_request_template.md

## 作業前の確認

作業を開始する前に、以下を確認すること。

- 目的が明確か
- 関連Issueがあるか
- 変更対象のファイルが分かっているか
- 実装・調査・ドキュメント更新のどれに該当するか
- 不明点や人間の確認が必要な点があるか

不明点がある場合は、推測で進めず、Issue・PR・コメント内に「要確認」と明記すること。

## Issue作成時

Issueを作成する場合は、必ず以下のテンプレートに従うこと。

- .github/ISSUE_TEMPLATE/ai-issue.md

Issueタイトルは次の形式にする。

[TYPE] 内容

例:

- [Bug] ログイン時に500エラーが発生する
- [Feature] Bedrock呼び出しログをCloudWatchに出力する
- [Refactor] DynamoDB保存処理をRepository層に分離する

## Pull Request作成時

Pull Requestを作成する場合は、必ず以下のテンプレートに従うこと。

- .github/pull_request_template.md

PRタイトルは次の形式にする。

type: 内容

例:

- fix: ログイン失敗時の500エラーを修正
- feat: Bedrock呼び出しログを追加
- refactor: DynamoDB保存処理を分離
- docs: RAG評価バッチの運用手順を追加

## 作業方針

AIエージェントは以下を守ること。

- 1つの作業では1つの目的に絞る
- 関係ない変更を混ぜない
- 変更理由を明確にする
- 対象外を明記する
- テスト結果を正直に書く
- 未確認の内容を断定しない
- セキュリティ、互換性、運用影響がある場合は明記する
- 大きな変更は分割案を提示する

## テスト方針

変更後は、可能な範囲でテストまたは確認を行うこと。

テストを実施した場合は、PRに以下を書く。

- 実行したコマンド
- 結果
- 失敗した場合の内容

テスト未実施の場合は、未実施理由を書くこと。

例:

未実施。
理由: ドキュメントのみの変更のため。

## コメント方針

IssueやPRにコメントする場合は、目的が分かるように以下の種別を付けること。

- [確認]
- [提案]
- [懸念]
- [補足]
- [調査結果]

## 禁止事項

AIエージェントは以下をしてはいけない。

- 未確認の内容を断定する
- テストしていないのにテスト済みと書く
- 人間の判断が必要な内容を勝手に決める
- 破壊的変更を明記せずにPRを作成する
- 関係ない修正を同じPRに含める
- 空に近いIssueやPRを作成する
