## Q1 比較検討: Rust + ratatui vs Go + bubbletea

### 比較表

| 観点 | Rust + ratatui | Go + bubbletea |
|------|---------------|----------------|
| **TUI表現力** | 非常に高い。低レベル描画制御が可能で、AAキャラクターやカスタムウィジェットの自由度が高い | 高い。Elm Architectureベースで宣言的UI。Lip Gloss（スタイリングライブラリ）との組み合わせが強力 |
| **AWS SDK** | aws-sdk-rust（公式、GA済み）。非同期対応（tokio）で高性能 | aws-sdk-go-v2（公式、成熟）。実績豊富で安定性が高い |
| **Bedrock連携** | aws-sdk-rust で Bedrock Runtime の InvokeModel を呼び出し可能 | aws-sdk-go-v2 で Bedrock Runtime の InvokeModel を呼び出し可能。Go の方がサンプルコードが豊富 |
| **バイナリ配布** | シングルバイナリ。クロスコンパイル可能だがターゲットごとの設定が必要 | シングルバイナリ。クロスコンパイルが非常に簡単（GOOS/GOARCH指定のみ） |
| **npm配布との相性** | バイナリをnpmパッケージに同梱するパターン（postinstallでDL）。やや複雑 | 同上。Go も Rust も同じアプローチが必要 |
| **開発速度** | 学習コストが高い。所有権・ライフタイムの理解が必要。ただし安全性は高い | 学習コストが低い。シンプルな言語仕様で素早くプロトタイプ可能 |
| **エコシステム** | ratatui は活発に開発中。ウィジェットが豊富 | bubbletea + bubbles + lip gloss のエコシステムが充実。コミュニティが活発 |
| **パフォーマンス** | 最高レベル。メモリ使用量も最小 | 十分高速。GCがあるがTUIアプリでは問題にならない |
| **日本語対応** | Unicode対応済み。全角文字の幅計算に注意が必要 | Unicode対応済み。同様に全角文字の幅計算に注意 |

### 推奨

**Go + bubbletea を推奨します。** 理由：

1. **開発速度**: devpetはゲーム性のあるTUIツールで、UIの試行錯誤が多い。Goの方が素早くイテレーションできる
2. **Elm Architecture**: bubbletea のモデル-更新-ビューパターンは、ペットの状態管理（HP、EXP、パラメータ）と相性が良い
3. **AWS SDK成熟度**: Go の AWS SDK はサンプルが豊富で、Bedrock連携の実装例も多い
4. **クロスコンパイル**: `GOOS=linux GOARCH=amd64 go build` のようにシンプル
5. **エコシステム**: bubbles（既製コンポーネント）+ lip gloss（スタイリング）で、画像のようなリッチなTUIを効率的に構築可能

ただし、Rustを選ぶ理由がある場合（Rustの学習目的、パフォーマンスへの強いこだわり等）はRustも十分実現可能です。

### Question 1-R

B) Rust + ratatui

## Q3 提案: ペットパラメータ設計

### パラメータ案

devpetの「人をダメにする」コンセプトと、AIペット育成の楽しさを両立するパラメータ設計を3案提示します。

### 案A: シンプル3パラメータ（画像ベース拡張）

| パラメータ | 説明 | AI応答への影響 | 進化への影響 |
|-----------|------|---------------|-------------|
| **論理** (Logic) | 論理的思考力 | 高い→技術的・分析的な返答 | 論理型進化ルートへ |
| **感情** (Emotion) | 感情の豊かさ | 高い→共感的・励まし系の返答 | 感情型進化ルートへ |
| **怠惰** (Laziness) | サボり度 | 高い→だるそう・やる気ない返答 | 怠惰型進化ルートへ |

- **増減ルール**: 「話しかける」で論理or感情がランダムに上昇、放置で怠惰が上昇
- **進化分岐**: 最も高いパラメータに応じて進化先が決まる

### 案B: 開発者テーマ4パラメータ

| パラメータ | 説明 | AI応答への影響 | 進化への影響 |
|-----------|------|---------------|-------------|
| **集中力** (Focus) | 開発への集中度 | 高い→的確で簡潔な返答 | エンジニア型進化 |
| **創造性** (Creativity) | アイデア力 | 高い→ユニークで面白い返答 | アーティスト型進化 |
| **社交性** (Social) | コミュ力 | 高い→おしゃべり・雑談好きな返答 | アイドル型進化 |
| **怠惰** (Laziness) | サボり度 | 高い→だるい・適当な返答 | ニート型進化 |

- **増減ルール**: 「話しかける」で集中力/創造性/社交性のいずれかが上昇（ランダム）、放置で怠惰が上昇
- **進化分岐**: 上位2パラメータの組み合わせで進化先が決まる（より多様な進化パターン）

### 案C: ミニマル2パラメータ + 隠しパラメータ

| パラメータ | 表示 | 説明 |
|-----------|------|------|
| **きずな** (Bond) | 表示 | ペットとの絆の深さ。話しかけると上昇、放置で低下 |
| **元気** (Energy) | 表示 | ペットの活力。時間経過で低下、話しかけると回復 |
| **性格値** (hidden) | 非表示 | 内部的に蓄積。話しかけた時間帯・頻度パターンで変動 |

- **シンプルさ**: ユーザーが意識するのは「きずな」と「元気」だけ
- **進化分岐**: 隠しパラメータ「性格値」で進化先が決まる（サプライズ要素）
- **AI応答**: きずなが高い→親密な返答、元気が低い→眠そうな返答

### 推奨

**案B（開発者テーマ4パラメータ）を推奨します。** 理由：
- devpetのターゲットが開発者なので、開発者テーマのパラメータが世界観に合う
- 4パラメータの組み合わせで進化パターンが豊富になり、育成の楽しさが増す
- 「怠惰」パラメータが「人をダメにする」コンセプトと直結
- 後から拡張しやすい（パラメータ追加が容易）

## Q6 比較検討: AWSバックエンド構成

### devpetに必要なバックエンド機能

まず、devpetで必要となるバックエンド機能を整理します：

| 機能 | 説明 | 必須度 |
|------|------|--------|
| **AI応答生成** | Claude Haiku を使ったペットの応答生成 | 必須 |
| **ユーザーデータ保存** | ペットの状態（レベル、パラメータ、進化段階等） | 必須 |
| **認証** | Cognito（オプション登録）でユーザー識別 | 必須 |
| **セーブデータ同期** | ローカル → クラウドへの手動セーブ | 必須 |
| **レート制限** | AI応答のAPI呼び出し制限（コスト管理） | 重要 |
| **ペット死亡判定** | 24時間放置の検知と死亡処理 | 重要 |

### 構成比較

#### 構成A: API Gateway + Lambda + DynamoDB（サーバーレス）

```
TUI Client
    |
    v
API Gateway (REST)
    |
    +---> Lambda (AI応答) ---> Bedrock (Claude Haiku)
    +---> Lambda (データ管理) ---> DynamoDB (ユーザー/ペットデータ)
    +---> Cognito (認証)
    +---> EventBridge (定期実行) ---> Lambda (死亡判定バッチ)
```

| 項目 | 評価 |
|------|------|
| **コスト** | 非常に低い。使った分だけ課金。小規模（〜100人）なら月数ドル程度 |
| **スケーラビリティ** | 自動スケール。将来の拡大にも対応 |
| **運用負荷** | ほぼゼロ。サーバー管理不要 |
| **開発速度** | 高い。Lambda関数を個別にデプロイ可能 |
| **コールドスタート** | Lambda のコールドスタートで初回応答が遅い場合あり（1-2秒） |
| **IaC** | CDK / SAM / Terraform で管理可能 |

#### 構成B: API Gateway + ECS/Fargate + RDS

| 項目 | 評価 |
|------|------|
| **コスト** | 高い。Fargate + RDS の常時稼働コストが発生（月$50〜） |
| **スケーラビリティ** | 手動/自動スケール設定が必要 |
| **運用負荷** | 中程度。コンテナ管理、DB管理が必要 |
| **開発速度** | 中程度。コンテナビルド・デプロイのパイプラインが必要 |
| **レスポンス** | 常時稼働のためコールドスタートなし |

#### 構成C: AppSync (GraphQL) + Lambda + DynamoDB

| 項目 | 評価 |
|------|------|
| **コスト** | 低い。サーバーレスベース |
| **リアルタイム** | GraphQL Subscriptions でリアルタイム通信可能 |
| **複雑さ** | GraphQLスキーマ定義が必要。TUIクライアントからのGraphQL呼び出しがやや複雑 |
| **適合性** | リアルタイム通信が不要なdevpetにはオーバースペック |

### 推奨

**構成A: API Gateway + Lambda + DynamoDB（サーバーレス）を推奨します。** 理由：

1. **コスト最適**: 小規模（〜100人）で使った分だけ課金。月数ドルで運用可能
2. **運用ゼロ**: サーバー管理不要。個人開発に最適
3. **devpetとの相性**: リクエスト/レスポンス型のAPI（話しかける→AI応答）にサーバーレスが最適
4. **Bedrock連携**: Lambda から Bedrock の InvokeModel を直接呼び出し。シンプル
5. **死亡判定**: EventBridge + Lambda で定期バッチ実行が簡単
6. **将来の拡張**: ユーザー増加時も自動スケール

コールドスタートの懸念は、Provisioned Concurrency や SnapStart（Java）で対策可能ですが、devpetの用途では1-2秒の遅延は許容範囲と考えます。

## Q7 補足確認: 認証の段階的設計

Q7で「Cognito認証だが、TUI自体は未登録でも使用可能にしたい」とのことでした。以下の設計で認識は合っていますか？

### 想定する認証フロー

1. **未登録状態**: TUIインストール直後はローカルのみで動作。ペット育成・AI応答（ローカルキャッシュ or 匿名APIアクセス）が可能
2. **登録状態**: Cognitoでアカウント作成すると、クラウドセーブ・データ同期が有効になる

### Question 7-R
上記の認証フローで合っていますか？

A) はい、その通り。未登録でもローカルで遊べて、登録するとクラウド同期が使える
B) 未登録でもAI応答（Bedrock）は使えるようにしたい（匿名APIアクセス）
C) AI応答を使うには登録必須。ローカルのみの場合はAI応答なしで育成のみ
X) Other (please describe after [Answer]: tag below)

[Answer]: A クラウドとの同期はユーザーがセーブしたときに実施する。
